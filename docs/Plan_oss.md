# Plan: Replace Supabase Storage With Alibaba Cloud OSS

## Goal

Keep Supabase as the application database, but stop using Supabase Storage for large media files.

Target architecture:

```text
Vercel / Next.js API
  - validates admin/idol permissions
  - writes message rows to Supabase Postgres
  - generates short-lived OSS/CDN access URLs

Supabase Postgres
  - stores messages, idols, visitor-related message metadata
  - stores OSS object keys in media_path and motion_video_path

Alibaba Cloud OSS
  - stores images, GIFs, voice files, videos, and motion photo video files
  - bucket stays private

Alibaba Cloud CDN
  - accelerates OSS media access for China users
  - browser downloads media directly from CDN/OSS, not through Vercel
```

The main performance improvement comes from moving large file traffic away from Supabase Storage and Vercel proxy routes. Text/API traffic still goes through Vercel and Supabase.

## Non-Goals

- Do not replace Supabase Postgres in this phase.
- Do not rewrite the chat data model.
- Do not make media truly download-proof. Browser-visible media can always be saved by determined users.
- Do not store long-lived public media URLs in the database.
- Do not expose OSS AccessKey secrets to client components or browser JavaScript.

## Recommended Data Model

Continue using the existing columns:

```text
messages.media_path
messages.motion_video_path
```

Store OSS object keys, not full signed URLs:

```text
asw/image/2026-07-01/uuid.jpg
asw/motion/2026-07-01/uuid.jpg
asw/motion/2026-07-01/uuid_video.mp4
```

Avoid storing values like:

```text
https://cdn.example.com/asw/image/xxx.jpg?Expires=...&Signature=...
```

Reason: signed URLs expire. If they are stored in Supabase, old chat messages will break after expiration and the database would need constant URL refreshes.

## Security Model

OSS bucket should be private.

The browser should never receive:

- OSS AccessKey ID
- OSS AccessKey Secret
- RAM user credentials
- unrestricted bucket-level credentials

The browser may receive:

- short-lived signed read URLs
- short-lived CDN-authenticated URLs
- direct upload URLs only when generated by a trusted server route

Practical guarantees:

- Users cannot list bucket contents.
- Users cannot access random files without a valid signed URL.
- Expired links stop working.
- Hotlinking can be reduced with CDN URL authentication and Referer rules.

Practical limitations:

- If a user can view an image or play a video in the browser, they can still save it using screenshots, screen recording, devtools, cache extraction, or network capture.
- The system can prevent open public file hosting, but cannot guarantee absolute anti-download protection.

Simple rule:

```text
Private bucket + short-lived URL = access control.
It is not digital rights management.
```

## Alibaba Cloud OSS/CDN Reference Points

The following points come from Alibaba Cloud official OSS/CDN documentation and directly affect this migration design:

- OSS private objects require temporary authorization for browser access. Alibaba Cloud documents that objects in an OSS bucket are private by default, and a server can generate a presigned GET URL so someone without credentials can download or preview a specific object until the URL expires.
  - Reference: [Download an object by using a presigned URL (Node.js SDK)](https://help.aliyun.com/en/oss/developer-reference/download-objects-using-a-presigned-url-generated-with-oss-sdk-for-node-js)
  - Reference: [Using presigned URLs to download or preview files](https://help.aliyun.com/en/oss/user-guide/how-to-obtain-the-url-of-a-single-object-or-the-urls-of-multiple-objects)

- OSS also supports server-generated presigned upload URLs. This allows direct browser-to-OSS upload without exposing AccessKey credentials. Alibaba Cloud notes that the URL is valid for a configurable period and can be reused within that period, so the application must generate unique object keys and avoid letting repeated PUTs overwrite an existing object.
  - Reference: [Upload an object using a presigned URL (Node.js SDK)](https://help.aliyun.com/en/oss/developer-reference/upload-objects-using-a-signed-url-generated-with-oss-sdk-for-node-js)
  - Reference: [Direct client upload to OSS](https://help.aliyun.com/en/oss/user-guide/uploading-objects-to-oss-directly-from-clients/)

- OSS + CDN is an official acceleration pattern. Alibaba Cloud recommends CDN acceleration for OSS and describes hotlink protection through Referer rules and URL authentication. For private buckets behind CDN, do not assume the original OSS signed URL model is still the only protection layer; CDN URL authentication should be configured for the accelerated domain.
  - Reference: [Accelerate OSS access with CDN](https://help.aliyun.com/en/oss/user-guide/cdn-acceleration)
  - Reference: [FAQ about CDN acceleration for OSS](https://help.aliyun.com/en/cdn/use-cases/cdn-acceleration-oss-faq)

- Referer hotlink protection can reduce unauthorized third-party embedding, but it is not strong identity authentication. Alibaba Cloud describes it as checking the HTTP `Referer` header against a whitelist or blacklist.
  - Reference: [Hotlink protection using OSS SDK for Node.js](https://help.aliyun.com/en/oss/developer-reference/nodejs-hotlink-protection)

- Large files and media playback should preserve byte range behavior. Alibaba Cloud OSS performance guidance recommends HTTP Range requests for large downloads, and its Range-related documentation expects valid `Content-Range` / `Content-Length` handling for partial content.
  - Reference: [OSS performance best practices](https://help.aliyun.com/en/oss/user-guide/oss-performance-best-practices/)
  - Reference: [OSS Range request error reference](https://help.aliyun.com/en/oss/user-guide/0041-00000103)

Implementation consequence:

```text
For normal display:
  Vercel signs -> browser downloads directly from Alibaba CDN/OSS.

For large direct uploads later:
  Vercel signs a unique PUT URL -> browser uploads directly to OSS.

For CDN production:
  Use CDN URL authentication and optional Referer protection.
  Do not rely on permanent public object URLs.
```

## Environment Variables

Add server-only variables:

```text
ALIYUN_OSS_REGION=oss-cn-hangzhou
ALIYUN_OSS_BUCKET=bubble-media
ALIYUN_OSS_ACCESS_KEY_ID=
ALIYUN_OSS_ACCESS_KEY_SECRET=
ALIYUN_OSS_INTERNAL_ENDPOINT=
ALIYUN_OSS_PUBLIC_ENDPOINT=
ALIYUN_CDN_BASE_URL=https://media.example.com
ALIYUN_CDN_AUTH_KEY=
MEDIA_STORAGE_PROVIDER=oss
```

Notes:

- Do not prefix secrets with `NEXT_PUBLIC_`.
- `ALIYUN_CDN_BASE_URL` should be used for browser-facing media URLs if CDN is enabled.
- `ALIYUN_OSS_PUBLIC_ENDPOINT` can be used before CDN is configured.
- `ALIYUN_OSS_INTERNAL_ENDPOINT` is only useful if the server runs inside Alibaba Cloud. Vercel cannot use Alibaba internal network endpoints.
- Add these keys to `.env.example` (with empty values) at the same time they are added to
  `lib/objectStorage.ts` config validation, matching the repo's existing convention of
  keeping `.env.example` in sync with real env vars.

Bucket/credential setup checklist before writing code:

```text
1. Create a dedicated RAM sub-account (not the main account AccessKey) scoped to only
   this bucket, with a policy limited to the needed actions (PutObject/GetObject/
   DeleteObject/ListObjects on this bucket ARN only).
2. Confirm the bucket ACL is Private (not public-read), and re-check this after any
   console changes — bucket ACL can be flipped accidentally via the console UI.
3. Store the RAM AccessKey ID/Secret only in Vercel env vars (server-only), never commit
   to the repo, never prefix with NEXT_PUBLIC_.
```

## Storage Abstraction

Create a storage adapter instead of calling Supabase Storage directly from route handlers.

Suggested file:

```text
lib/objectStorage.ts
```

Suggested API:

```ts
export type UploadObjectInput = {
  key: string;
  file: File | Blob | Buffer;
  contentType?: string;
};

export type SignedReadUrlInput = {
  key: string;
  expiresInSeconds: number;
  contentTypeHint?: string;
};

export async function uploadObject(input: UploadObjectInput): Promise<void>;
export async function deleteObjects(keys: string[]): Promise<void>;
export async function createSignedReadUrl(input: SignedReadUrlInput): Promise<string>;
export async function createSignedUploadUrl(key: string, expiresInSeconds: number): Promise<string>;
```

The business routes should not know whether the object backend is Supabase Storage, OSS, COS, or another provider.

## Upload Flow

Current flow:

```text
browser -> Vercel /api/idol/upload -> Supabase Storage -> Supabase messages row
```

Target flow:

```text
browser -> Vercel /api/idol/upload -> OSS putObject -> Supabase messages row
```

For normal media:

```text
1. Validate idol/admin session.
2. Validate file type and size.
3. Generate object key:
   {idolId}/{messageType}/{yyyy-mm-dd}/{uuid}.{ext}
4. Upload file to OSS.
5. Insert message row in Supabase with media_path = object key.
6. If database insert fails, delete the uploaded OSS object.
```

For motion photos:

```text
1. Validate still image and motion video.
2. Generate shared base object key.
3. Upload still image to OSS.
4. Upload motion video to OSS.
5. Insert message row with:
   type = motion
   media_path = still image key
   motion_video_path = video key
6. If any later step fails, delete already uploaded OSS objects.
```

## Read Flow

Current direct signed URL flow:

```text
ChatBubble -> /api/media/signed-url -> Supabase createSignedUrl -> browser loads Supabase URL
```

Target direct CDN/OSS flow:

```text
ChatBubble -> /api/media/signed-url -> validate message in Supabase -> generate OSS/CDN signed URL -> browser loads CDN/OSS URL
```

The media bytes should not pass through Vercel for the common display path.

This gives the best China-user performance:

```text
browser -> Alibaba Cloud CDN/OSS
```

instead of:

```text
browser -> Vercel -> Supabase Storage
```

## Proxy Read Flow

The existing route:

```text
app/api/media/file/route.ts
```

can remain as a fallback for cases where same-origin proxy behavior is needed.

For best performance, images and motion preview videos should prefer `/api/media/signed-url` and browser-direct CDN/OSS loading.

If proxy support is retained:

```text
1. Validate media_path exists in messages.
2. Generate short-lived OSS URL.
3. Forward Range header for video/audio.
4. Stream OSS response back to browser.
```

Important: proxying large files through Vercel costs performance and bandwidth. Use it only where direct signed URLs are not acceptable.

## Delete Flow

Current flow:

```text
delete message -> remove Supabase Storage objects -> delete Supabase row
```

Target flow:

```text
delete message -> remove OSS objects -> delete Supabase row
```

Recommended behavior:

```text
1. Fetch message by id and idol_id/admin scope.
2. Collect media_path and motion_video_path.
3. Delete corresponding OSS objects.
4. Delete Supabase row.
```

If OSS delete fails, return an error and do not delete the row. This avoids orphaning a database row that points to an existing private file.

Current direct `supabase.storage.remove(...)` call sites that must all move to
`deleteObjects()` together (partial migration would leave some deletes hitting Supabase
Storage and others hitting OSS):

```text
app/api/idol/messages/route.ts       (DELETE handler, message media cleanup)
app/api/idol/profile/route.ts        (avatar replace/delete cleanup)
app/api/idol/upload/route.ts         (rollback cleanup on failed DB insert)
```

## OSS SDK Notes

Use Alibaba Cloud's Node.js OSS SDK or official signing utilities from server-only code.

Expected capabilities:

- `put` / `putObject` for server-side uploads
- `delete` / `deleteMulti` for object deletion
- signed read URL generation
- signed upload URL generation if direct browser upload is needed later

For the first implementation, server-side upload through Vercel is simpler and matches the current API shape.

Potential limitation: uploading very large files through Vercel can hit request body size or function timeout limits. If large videos become common, move to direct browser-to-OSS upload with a server-generated presigned upload URL.

## CDN Signing Strategy

Preferred production setup:

```text
Private OSS bucket -> Alibaba Cloud CDN custom domain -> CDN URL authentication
```

Recommended:

- Use a dedicated media domain, for example `media.example.com`.
- Keep OSS bucket private.
- Enable CDN URL authentication for expiring links.
- Optionally enable Referer hotlink protection.
- Configure reasonable cache headers for immutable object keys.

Because object keys use UUIDs and are never overwritten, CDN caching is safe.

Suggested cache policy:

```text
images/gifs: long cache, immutable object keys
voice/video: long cache if signed URL/auth rules allow it
```

## Image Transformation

Supabase Storage currently supports image transform options in signed URLs. OSS does not use the same API.

Replacement options:

1. Use OSS Image Processing query parameters for images.
2. Generate thumbnails during upload and store separate object keys.
3. Keep original images only and rely on existing client-side compression.

Recommended phase 1:

```text
Keep current client-side image compression.
Do not implement server-side image transformation unless real bandwidth data shows it is needed.
```

This is not optional cleanup — it is a functional regression if left unaddressed. Today,
`app/api/media/signed-url/route.ts`, `app/api/media/file/route.ts`, and
`app/api/media/avatar/route.ts` all pass `{ transform: { width, height, resize } }` to
Supabase's `createSignedUrl`. OSS has no equivalent option on its signed-URL call, so a
naive storage-adapter swap will make these three routes throw or silently return a
different shape. Phase 1 must explicitly decide one of:

```text
a) Drop the transform option entirely and return original-size objects
   (rely on existing client-side compression; avatar route falls back to full-size image).
b) Append OSS Image Processing query params (x-oss-process=image/resize,w_128,h_128)
   to the generated signed URL.
```

Pick (a) for the first cut unless avatar/thumbnail bandwidth is already a measured problem.

If thumbnails are added later:

```text
media_path = original object key
thumbnail_path = thumbnail object key
```

This would require a schema change.

## Migration Plan For Existing Media

Existing Supabase Storage files should be copied to OSS while preserving object keys.

Migration algorithm:

```text
1. Query Supabase messages where media_path or motion_video_path is not null.
2. Build a unique set of object keys.
3. For each key:
   a. Download object from Supabase Storage.
   b. Upload object to OSS with the same key.
   c. Verify size/content-type where possible.
4. Do not update message rows if keys are preserved.
5. Switch runtime storage provider to OSS.
6. Keep Supabase Storage files temporarily for rollback.
7. Delete Supabase Storage files only after production verification.
```

Rollback is simple if object keys are preserved:

```text
MEDIA_STORAGE_PROVIDER=supabase
```

or revert the code deployment.

## Implementation Phases

### Phase 1: Adapter And OSS Uploads

- Add OSS dependencies.
- Add server-only OSS config validation.
- Add `lib/objectStorage.ts`.
- Replace upload and delete calls in active idol routes.
- Keep Supabase database writes unchanged.

### Phase 2: Direct Signed Read URLs

- Replace `/api/media/signed-url` implementation.
- Make returned URLs point to OSS/CDN.
- Keep validation against Supabase messages before signing.
- Prefer direct browser loading for image, GIF, voice, video, and motion video.

### Phase 3: Legacy Admin Compatibility

- Update legacy admin upload/presign/record/delete routes.
- Remove direct Supabase Storage assumptions from route handlers.

### Phase 4: Migration

- Write a one-off migration script.
- Copy existing Supabase Storage objects to OSS.
- Verify old messages still load.
- Keep Supabase Storage as backup until confidence is high.

### Phase 5: CDN Hardening

- Configure custom CDN domain.
- Enable CDN URL authentication.
- Add Referer hotlink protection if appropriate.
- Tune cache headers.

## Testing Checklist

Upload and display:

- image
- GIF
- voice message
- video
- motion photo still image
- motion photo video playback

Access control:

- signed URL expires correctly
- random object key cannot be opened without signing
- deleted message media is no longer accessible after object deletion
- browser never receives OSS secrets

Performance:

- media URL host is Alibaba CDN/OSS, not Supabase
- large files do not stream through Vercel on the common path
- video/audio Range requests work when proxy route is used

Regression:

- public admin messages still show for visitors
- private user messages still show only in the intended visitor thread
- idol console still shows own public messages and fan private messages
- delete still removes both database row and media object

## Main Risk Points

1. Signed URL and CDN auth mismatch can cause media to work locally but fail behind CDN.
2. Motion photo has two objects; cleanup must handle partial upload failures.
3. Vercel server-side upload may be unsuitable for large videos.
4. Image transformation behavior differs between Supabase Storage and OSS.
5. If full signed URLs are stored in Supabase, old messages will break after expiration.

## Recommended Decision

Use this design:

```text
Supabase Postgres remains the source of truth for messages.
OSS stores object bytes.
Supabase stores only OSS object keys.
Vercel signs short-lived OSS/CDN URLs after checking message existence.
Browser loads large media directly from Alibaba Cloud CDN/OSS.
```

This keeps the current product logic stable while moving the slowest traffic to infrastructure that is closer to China users.
